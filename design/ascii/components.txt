-------------------------------------------------------------------------------
DIRECTIONS FOR AGENTS CONSUMING THIS FILE
-------------------------------------------------------------------------------
This file is the AskeeDS ASCII design system and component library (canonical
location: design/ascii/components.txt). By Spencer Goldade. Read it when
implementing or modifying terminal UI, layouts, panes, or text-based components
for any TUI using AskeeDS.

HOW TO PARSE
  - Component start: line beginning with three U+241F (␟) then "COMPONENT: "
    followed by the component name (e.g. layout.app.shell, room-card.default).
  - Meta: lines beginning with one ␟ then a key and value (e.g. ␟ props: ...,
    ␟ description: ..., ␟ component-status: ...). Meta applies to the component
    that follows the previous component boundary. When status is Deprecated,
    include ␟ replaced-by: <component name>.
  - Multi-line meta: a line that begins with one ␟ followed by two or more
    spaces (e.g. ␟   continuation text) appends to the previous meta key's
    value with a newline. Use this for purpose, default-view, prop-types, or
    edge-cases when one line is not enough.
  - Optional intent meta (documentation alongside the component): ␟ purpose:,
    ␟ default-view:, ␟ randomized-view:, ␟ prop-types:, ␟ edge-cases: capture
    design intent for implementers and tooling (e.g. component visual test).
  - Everything after the meta block, until the next ␟␟␟ line, is the
    component's ASCII art (structure and example content). A single blank line
    between the last meta line and the first art line is recommended for
    readability and is treated as part of the art block.
  - Do not use the character ␟ (U+241F) inside ASCII art. Parsers split on ␟
    to find component boundaries and meta lines; a ␟ in the art would break
    parsing. Use only box-drawing and content characters in the art block.

HOW TO USE
  - Look up a component by name when building or styling a TUI element. Use
    the ASCII art as the structural reference (borders, spacing, layout).
  - Border characters (| - + etc.) are defined in design/ascii/box-drawing.yaml;
    load from there in code rather than hardcoding. This file shows the
    intended shape only.
  - Semantic colors are applied in code (e.g. Rich/Textual) using your
    project's color tokens; keep surfaces, text, and accents consistent.
    (Rich/Textual) per ␟ color-hint: on each component when present.
  - Respect props listed in meta (e.g. title, description_text, directions[])
    when implementing; they define the data the component expects.
  - Max width 80 chars; prefer 60–70 for body text. Test at 80×24; 256-color.
  - Layout components (e.g. layout.two-column, layout.stack): Props like
    left_content, right_content, and blocks[] are content slots. Your runtime
    may pass pre-rendered strings or the output of other components; this file
    does not define a composition API. Ensure what you pass respects max width
    (80 chars) and box-drawing.

HOW TO ADD A NEW COMPONENT
  - Pick a name:
    - Use dot notation (category.variant), e.g. layout.app.shell, menu.main.
    - Keep it stable; avoid reusing the same name for different layouts.
  - Add a component block:
    - Start with a boundary line: ␟␟␟ COMPONENT: <name>
    - Add meta lines: at minimum ␟ description: and ␟ props:
    - Include ␟ component-status: (see COMPONENT STATUS below).
    - Optionally add ␟ shape:, ␟ variant:, ␟ interactive:, ␟ color-hint:
  - Sketch the ASCII art:
    - Use only box characters from box-drawing.yaml for borders.
    - Stay within 80 characters per line; prefer 60–70 for readability.
    - Do not use the ␟ character anywhere in the art block.
  - Update contracts:
    - If you introduce new shape patterns, reflect them in PROP SHAPES below.
    - Update project documentation only if the component
      changes higher-level flows or concepts.
  - Keep manifest in sync:
    - Run `python3 tools/update_manifest.py` (or `npm run update:manifest`) to
      regenerate design/ascii/manifest.yaml from components.txt so tools and
      tests can verify coverage.

PROPS CONTRACT
  - Naming: Prop names use snake_case (e.g. hp_current, body_text). Refer to
    props by name (e.g. props["title"], keyword arguments, or your runtime's
    equivalent).
  - Optionality: If a prop is optional, the component's ␟ props line may suffix
    it with _optional or list optional props on a separate ␟ optional: line.
    Props without that marking are required unless the description says otherwise.
  - Arrays: A prop named with trailing [] (e.g. items[]) denotes a list. The
    shape of each element is defined in PROP SHAPES below or in the component's
    ␟ shape: line.
  - Single source of truth: When implementing, treat the component's ␟ props:
    and any ␟ shape: (or PROP SHAPES entry) as the contract for what to pass in.
    Tooling that generates default or randomized test props (e.g. the component
    visual test) may read design/ascii/prop_shapes.yaml for array element shapes.
  - Color: Every component accepts an optional prop color_role (or color_role_optional).
    A semantic role id (e.g. status.good, ui.accent) that the implementation maps to
    its own tokens. When absent, use the component's ␟ color-hint: default(s) or theme.

PROP SHAPES
  - room-card.default: entity_list: { id, label }[] (items/NPCs in room).
    exits: string[] (e.g. ["north","south"]).
  - entity-list.room: items: { id, label }[], npcs: { id, label }[].
  - minimap.default: grid: 2D array of role ids or characters (per map-tiles.yaml).
    legend_entries: { char, label }[]. player_position: { row, col } or equivalent.
  - layout.two-column: left_content, right_content: pre-rendered string or
    runtime's child content (see composition note in HOW TO USE).
  - layout.stack: blocks[]: list of pre-rendered strings or child content.
  - tracker.objective: objectives: { id?, label, checked: bool }[].
  - inventory.grid: slots: { id, label?, item_ref? }[]. columns: number.
  - inventory.list: items: { id, label, quantity? }[].
  - character-sheet.compact: stats: { label, value }[] or { label, current, max }[]
    for bar-style rows.
  - tree.compact: nodes: { id, label, children?: nodes }[] (nested) or
    { id, label, depth }[] (flat with indent).
  - breadcrumb.inline: segments: { id, label }[] (ordered path from root to current).
  - choice-wheel.inline: options: { id, label }[].
  - quick-select.radial: options: { id, label }[]. selected_id: id of selected.
    Positions 1-4 (or 1-8) map clockwise from top: 1=top, 2=right, 3=bottom, 4=left.
  - nav.vertical: items: { id, label }[]. active_id: id of the active item.
  - menu.main / menu.pause: items: { id, label }[].
  - status-icon.row: icons: { symbol, label? }[] or { id, symbol, label? }[].
  - screen.death: actions: { id, label }[] (e.g. Respawn, Main menu).
  - decoration.placeholder: width, height: character-cell dimensions of the slot;
    art is cropped or centered to fit. Content is a separate asset or catalog entry.
  - panel.survival-status: needs: { label, current, max }[].
  - tree.relationships: relations: { name, attitude, tags: string[] }[].
  - screen.crafting: inputs: { label, item_name?, quantity? }[]; resource_costs:
    { label, amount }[].
  - cooldown.row: abilities: { label, turns_left }[].
  - panel.consequence: body_conditions: { label, severity }[]; mental_conditions:
    { label, severity }[].
  - All components: color_role_optional: string (semantic role id for implementation
    to map to project color tokens; when absent, use ␟ color-hint: or theme).

COLOR-HINT BEHAVIOR
  - HP / resource bands (status.good, status.warning, status.danger): Use good
    when value is above ~70% of max, warning between ~30% and ~70%, danger below
    ~30%. Exact thresholds are implementation-defined; document the intent.
  - "Entity for others" (e.g. mana, stamina): Use your project's color
    tokens for non-HP resources (e.g. entity.item, ui.accent) as appropriate;
    do not rely on color alone for meaning.

GUIDELINES FOR ASCII DESIGN SYSTEM COMPONENTS (authors / editors)
-------------------------------------------------------------------------------
Source of truth: design/ascii/ (this file, box-drawing.yaml, map-tiles.yaml,
manifest.yaml). See design/ascii/README.md.

PATTERNS AND USAGE
  - Prefer composing components (e.g. modal = overlay + card + buttons). Use
    ␟ pattern: / ␟ use-case: to link components to flows.
  - Variants: When a component lists ␟ variant: (e.g. tooltip.default with
    default | stats) or when two components share the same structure (e.g.
    menu.main and menu.pause), implement as one parameterized component when
    the layout is identical; use the variant or use-case to change props or
    copy only. Document which props apply to which variant in the component's
    ␟ props: / ␟ optional: lines.

ACCESSIBILITY AND CONTENT
  - Structure (headings, lists) supports screen readers; keep order and labels
    consistent. Color is applied in code per your project's tokens; do not
    rely on shape alone for meaning.
  - Content style: Short, present tense; noun-based labels. Match
    Content style: short, present tense; noun-based labels. Avoid jargon.
  - Brand: One consistent box-drawing set (box-drawing.yaml); avoid mixing
    decorative Unicode in structure. Decorative ASCII art in a dedicated slot
    (decoration.placeholder) uses ASCII-only characters.
  - Empty and error states: When a component has no data (e.g. empty inventory,
    no objectives, failed load), show a single-line placeholder message or the
    component structure with one line such as "(none)" or "No items." Avoid
    leaving the component area blank. For errors, use feedback.error or a
    dedicated line with status.danger.

DELIMITERS
  - ␟␟␟ COMPONENT: <name>   Start of a new component (U+241F SYMBOL FOR UNIT SEPARATOR).
  - ␟ <key>: <value>         Meta line: description, props, variants, etc.
  One ␟ = meta; three ␟ = component boundary. No other uses of ␟ in art.

COMPONENT STATUS
  - Each component has ␟ component-status: with one of: Ideated, To Do,
    In Progress, In Review, Approved, Cancelled, Deprecated.
  - Ideated: Idea only; not yet decided to pursue.
  - To Do: Decided to pursue; not started.
  - In Progress: Currently being worked on.
  - In Review: Being tested or reviewed for quality.
  - Approved: Passed review; safe to use in projects.
  - Cancelled: Decided not to pursue.
  - Deprecated: Replaced by another component. When Deprecated, you must also
    add ␟ replaced-by: <component.name> (the component that replaces this one).
  - Status need not follow a linear progression (e.g. Approved → Deprecated,
    or skip statuses). Place ␟ component-status: near the top of the meta block.

NAMING
  - Use dot notation: category.variant (e.g. layout.app.shell, button.icon).
  - Noun-based, present tense; avoid jargon. Matches design tokens (e.g.
    status.good, entity.item).

BOX-DRAWING AND WIDTH
  - Use characters from design/ascii/box-drawing.yaml for borders. Do not
    hardcode alternate box glyphs here; change the YAML and keep this file
    consistent with the active set (e.g. light: | - +).
  - Max line length 80 chars; aim 60–70 for readable blocks (per ui-ux §5.1).
  - Test at 80×24 min; 256-color safe. No truecolor-only or wrapping.

SPACING AND HIERARCHY
  - One space between border and inner text (padding). Keep vertical
    rhythm consistent (blank line between sections when the component has
    distinct regions). Keep primary vs secondary visually distinct.

PROPS AND META
  - List props on ␟ lines: ␟ props: name, label, icon. Optional: ␟ variant:,
    ␟ description:, ␟ shape:. Code will use these when implementing components.
  - Interactivity: Optional meta ␟ interactive: true indicates that the
    component is focusable or emits events (e.g. button, menu item, choice
    option). Use for accessibility and focus order; this file does not define
    event names or key bindings.

COLOR
  - This file defines structure only. Semantic color lives in
    semantic color tokens and is applied in code (Rich/Textual). You may
    add ␟ color-hint: <token> for documentation (e.g. ␟ color-hint: ui.accent
    for border).

ADDING COMPONENTS
  - Check existing components to avoid duplicates. Use the same ␟/␟␟␟
    convention. Add at least ␟ description:, ␟ props: (if any), and
    ␟ component-status: (see COMPONENT STATUS above). Keep examples minimal
    and representative. Add the component name to design/ascii/manifest.yaml.

␟␟␟ COMPONENT: layout.app.shell
␟ description: Main app shell with header, sidebar, and content area
␟ component-status: In Review
␟ usage: Use as the top-level frame for the whole TUI, wrapping nav and main content.
␟ props: color_role_optional
␟ purpose: The app shell is the root-level layout container for the entire game UI.
␟   It acts as the structural skeleton that all other components live inside
␟   — defining the overall page chrome, background, and visual theme. In
␟   game engines like Unity or Godot, this is analogous to the root Canvas
␟   or the master scene node. In web-based game UIs (e.g. Twine, Bitsy,
␟   browser RPGs), it sets the viewport constraints, background color or
␟   texture, and font defaults.
␟ default-view: Without a `color_role`, the shell renders with a neutral dark background
␟   (near-black), filling the full viewport. It creates a centered or
␟   full-bleed container that child components slot into. No visible
␟   elements of its own — purely structural. Typically renders as a `<div>`
␟   with `width: 100vw; height: 100vh; overflow: hidden`.
␟ randomized-view: Changing `color_role` (e.g. "danger", "arcane", "nature") shifts the
␟   global palette — background color, border accent color, and text color
␟   all derive from the role. The layout itself (dimensions, flex direction)
␟   stays identical; only the visual theme changes. This allows a single
␟   shell to feel dramatically different across game zones or modes.
␟ prop-types: - `color_role_optional`: `string | undefined` — A semantic theme token
␟   such as `"neutral"`, `"danger"`, `"arcane"`, `"nature"`, `"frost"`. Maps
␟   to a predefined color palette. If omitted, defaults to `"neutral"` or
␟   the system default theme.
␟ edge-cases: - No `color_role` provided → falls back to default theme without
␟   throwing. - Unknown `color_role` string → should log a warning and fall
␟   back gracefully. - Very small viewport (mobile, 320px wide) → must not
␟   overflow or clip child components. - Used inside an iframe or embedded
␟   context → needs to respect parent container dimensions rather than
␟   `100vw/100vh`.

␟␟␟ COMPONENT: button.icon
␟ description: Button with leading icon (one or two chars).
␟ component-status: Approved
␟ usage: Use for primary or secondary actions where a small symbol reinforces meaning.
␟ props: label, icon, color_role_optional
␟ purpose: A compact, icon-first button used for game actions, toolbar controls,
␟   and HUD shortcuts (e.g. "Open Inventory", "Cast Spell", "Save Game").
␟   Common in RPG action bars, strategy game control panels, and system
␟   menus. The label provides an accessible text name and may display as a
␟   tooltip or screen-reader-only text.
␟ default-view: Renders as a square or near-square button with the icon centered inside
␟   it. The label is visually hidden (or shown on hover as a tooltip) to
␟   preserve compact layout. Default `color_role` produces a neutral/muted
␟   button; hover and active states provide visual feedback. Border radius
␟   and sizing should be consistent with the design system's base unit (e.g.
␟   8px radius, 40×40px).
␟ randomized-view: - Changing `icon` (e.g. a sword → a shield → a potion flask) changes the
␟   glyph rendered, but button dimensions stay fixed. - Changing
␟   `color_role` (e.g. `"danger"` → red tint, `"success"` → green tint)
␟   recolors the button's background/border/icon tint. - Changing `label`
␟   doesn't affect visual appearance unless a tooltip is shown, but updates
␟   accessible name.
␟ prop-types: - `label`: `string` — Short human-readable action name, e.g. `"Attack"`,
␟   `"Open Map"`, `"Drink Potion"`. Used for accessibility and tooltip. -
␟   `icon`: `string` — Icon identifier, e.g. a Unicode glyph (`"⚔"`, `"🗺"`),
␟   an icon library key (`"sword"`, `"map"`), or an image path
␟   (`"/icons/potion.svg"`). - `color_role_optional`: `string | undefined` —
␟   Theme token as above.
␟ edge-cases: - Very long `label` string used in tooltip → must truncate or wrap
␟   gracefully. - `icon` is an invalid path or unknown key → show a fallback
␟   placeholder glyph. - Icon renders at very small sizes → must remain
␟   legible (minimum 16px target size for accessibility). - `color_role`
␟   that maps to a very light color → ensure icon contrast ratio meets
␟   accessibility minimums.

␟␟␟ COMPONENT: tooltip.default
␟ description: Bordered tooltip; optional stats variant.
␟ component-status: In Review
␟ usage: Show contextual help or quick stats when the user focuses or hovers an element.
␟ props: text; optional variant stats: name, stats_dict, color_role_optional
␟ variant: default | stats
␟ shape: box.single
␟ purpose: Tooltips are contextual information overlays that appear on hover or
␟   focus. In games, they reveal item descriptions, stat breakdowns, ability
␟   costs, or lore entries. The stats variant (with `name` + `stats_dict`)
␟   is specifically for game items or characters — showing a mini stat sheet
␟   without navigating away. Common in Diablo-style ARPGs, Civilization, and
␟   most card games (e.g. Hearthstone, Slay the Spire).
␟ default-view: The default (text-only) variant renders as a small floating panel near
␟   the hovered element, containing a single line or short paragraph of
␟   descriptive text. It should appear with a short delay (~300ms) and
␟   disappear when the cursor moves away. Default styling: dark background,
␟   light text, subtle border or shadow, 12–14px font.
␟   The stats variant shows a header with `name` (item/character name),
␟   followed by a key-value breakdown from `stats_dict` (e.g. "ATK: 12",
␟   "DEF: 5", "Weight: 2kg"). This variant is taller and may include a
␟   divider between name and stats.
␟ randomized-view: - Swapping `text` changes only the content. Layout stays identical. -
␟   Populating `stats_dict` with many entries (10+) forces the tooltip to
␟   scroll or expand — its width should remain capped. - Changing
␟   `color_role` recolors the tooltip panel to match a theme (e.g. `"rare"`
␟   = blue, `"legendary"` = gold), a pattern from games like World of
␟   Warcraft item quality coloring.
␟ prop-types: - `text`: `string` — Descriptive prose, e.g. `"A rusted blade, worn by
␟   time but still sharp enough to cut."` Can be 1–3 sentences. - `name`:
␟   `string | undefined` — Entity name for stats variant, e.g. `"Iron
␟   Sword"`, `"Guard Captain"`. - `stats_dict`: `Record<string, string |
␟   number> | undefined` — Key-value pairs, e.g. `{ "ATK": 12, "Weight":
␟   "2.4kg", "Rarity": "Common" }`. - `color_role_optional`: `string |
␟   undefined` — Theme token.
␟ edge-cases: - `text` is very long (paragraph) → needs max-width and text wrapping;
␟   avoid overflow. - `stats_dict` is empty `{}` → stats variant should hide
␟   the stats section or fall back to text-only display. - Tooltip
␟   positioned near edge of viewport → must flip direction (above/below,
␟   left/right) to stay in view. - Touch devices have no "hover" state →
␟   tooltip should appear on tap/focus and dismiss on second tap. - `text`
␟   and `stats_dict` both provided → clarify which takes visual priority.

␟␟␟ COMPONENT: table.fourcolumn
␟ description: Four-column bordered table; header row and data rows.
␟ component-status: In Review
␟ usage: Use for structured tabular data such as stats, settings, or comparisons.
␟ props: label, content, color_role_optional
␟ purpose: A four-column data table for presenting structured game data — loot
␟   tables, skill comparisons, leaderboard entries, equipment stats, quest
␟   logs, or bestiary entries. The rigid four-column layout suits data that
␟   naturally fits a left-aligned label + 3 data columns pattern, such as
␟   "Stat / Base / Modified / Max" or "Item / Weight / Value / Count".
␟ default-view: Renders as a bordered table with a header row (derived from `label`) and
␟   data rows from `content`. Alternating row shading (zebra stripes) aids
␟   readability. Column widths distribute evenly unless overridden. The
␟   `label` appears as a table caption or header area above the grid.
␟ randomized-view: - More rows in `content` → table grows vertically; scrollable if
␟   exceeding a container height. - More columns used differently (sparse
␟   data, empty cells) → cells show an em dash or "—" for missing values. -
␟   `color_role` changes header and border colors — e.g. `"arcane"` might
␟   give a purple header row.
␟ prop-types: - `label`: `string` — Table title/caption, e.g. `"Item Comparison"`,
␟   `"Skill Tree Stats"`. - `content`: `Array<[string, string, string,
␟   string]>` or `Array<Record<string, string>>` — Array of row data, each
␟   with exactly 4 values. Could also be structured as `{ headers: string[],
␟   rows: string[][] }`. - `color_role_optional`: `string | undefined`.
␟ edge-cases: - Fewer than 4 values in a row → pad with empty strings or "—". - Cell
␟   content is very long text → truncate with ellipsis or allow wrapping. -
␟   Zero rows in `content` → show an empty state message ("No data
␟   available"). - Non-string values (numbers, booleans) → should coerce to
␟   string for display. - Very narrow viewport → consider collapsing to a
␟   card-per-row layout on mobile.

␟␟␟ COMPONENT: status-bar.default
␟ description: Single-line HUD; HP, location, turn. Top or bottom of screen.
␟ component-status: In Review
␟ usage: Pin to the top or bottom of the screen to show core moment-to-moment status.
␟ props: hp_current, hp_max, location_name, turn_count, color_role_optional
␟ color-hint: status.good/warning/danger for HP band; ui.muted for labels
␟ purpose: The core game HUD status bar — a persistent top or bottom bar showing
␟   the player's current health, their location, and the current turn
␟   number. This is a fundamental UI pattern found in virtually every RPG,
␟   roguelike, and strategy game (Nethack, XCOM, Stardew Valley, Darkest
␟   Dungeon). It gives the player at-a-glance situational awareness without
␟   navigating menus.
␟ default-view: A horizontal bar divided into regions: a health indicator on the left
␟   (typically showing `hp_current / hp_max` as both a number and a colored
␟   fill bar), the `location_name` centered, and `turn_count` on the right.
␟   The HP bar fills proportionally — green at full health, transitioning to
␟   yellow then red as HP drops.
␟ randomized-view: - `hp_current` changing from 100 to 3 → bar shrinks and color shifts
␟   from green to critical red. - `location_name` switching between short
␟   ("Cave") and long ("The Ancient Sunken Vault of Malachar") → long names
␟   must truncate with ellipsis. - `turn_count` incrementing → counter
␟   updates cleanly; at large numbers (9999+) check for layout overflow. -
␟   `color_role` → changes accent color of the bar's border/background, not
␟   the HP fill color (which is semantic).
␟ prop-types: - `hp_current`: `number` (non-negative integer) — Current hit points.
␟   E.g. `45`. - `hp_max`: `number` (positive integer) — Maximum hit points.
␟   E.g. `100`. - `location_name`: `string` — Current room or zone name.
␟   E.g. `"Goblin Warren"`, `"Floor 3 — East Corridor"`. - `turn_count`:
␟   `number` (non-negative integer) — Current game turn. E.g. `142`. -
␟   `color_role_optional`: `string | undefined`.
␟ edge-cases: - `hp_current > hp_max` (e.g. buffed/overheal) → bar should cap at full
␟   or show overflow indicator. - `hp_current = 0` → bar is empty; consider
␟   triggering a "death" visual state. - `hp_max = 0` → guard against
␟   division-by-zero in percentage calculation. - Extremely long
␟   `location_name` → truncate or scroll. - `turn_count` overflows display
␟   space → abbreviate (e.g. "9.9k+").

␟␟␟ COMPONENT: status-bar.risk

␟ component-status: In Review
␟ usage: Swap in when the game tracks global danger level or player fortune alongside HP.
␟ props: hp_current, hp_max, location_name, turn_count, risk_level, luck_band_optional, color_role_optional
␟ color-hint: status.good/warning/danger for HP; entity.alert for high risk; ui.accent for luck
␟ purpose: An extended status bar for games with explicit risk/danger mechanics —
